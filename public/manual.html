Sherlock Manual

A guide to probabilistic chain-analysis in Sherlock.

1. Getting Started

Welcome to Sherlock, a local chain analysis engine. Sherlock is designed to parse raw Bitcoin Core block data, automatically identify behavioral patterns, and assign probabilistic tags and heuristic classifications to individual transactions.

Importing Blocks

To analyze a new block, locate the in (Import New Block) button on the fixed left sidebar. You will need three files corresponding to the same block from a Bitcoin Core datadir:

  • blk*.dat - The raw block binary data.
  • rev*.dat - The undo (reverse) data containing spent output information.
  • xor.dat - The XOR mask applied to the files by Bitcoin Core.

Click the in button, select the three corresponding files from your local system when prompted, and click "ANALYZE FILES". The backend engine will parse the raw binary, link the inputs, run all heuristics, and automatically load the results into your dashboard.

The Chain Data Browser (Dashboard)

The primary view of the Sherlock web interface is the Chain Data Browser, located at the bottom of the main page.

  • Selecting Data: Use the dropdown menu in the dashboard header to select a previously parsed and analyzed block file.
  • High-Level Stats: Once loaded, the top bar displays vital signs for the block: Total Transactions, Flagged Transactions, Average Fee rate, and total triggered Heuristics.
  • Block Breakdown: To the right, a dynamic pie chart visualizes the byte-allocation within the block, showing the ratio of standard transaction data vs. SegWit witness data vs. block header overhead.
  • Interacting with Transactions: The main list shows the most notable flagged transactions. You can click on any transaction row to open a detailed modal containing:
    • The transaction's unique hash (TXID).
    • A summary of exactly which privacy heuristics were triggered and why.
    • A breakdown of inputs (where the money came from) and outputs (where it went), including script types and values.

Transaction Tracer

  1. Click the tt (Transaction Tracer) button on the left sidebar.
  2. Paste a specific 64-character Transaction ID (TXID) into the text box.
  3. Click "TRACE". If the transaction exists in your loaded block, Sherlock will immediately open its detailed breakdown.

Information Glossary

  1. Click the ig (Information Glossary) button on the left sidebar.
  2. Scroll the list or use the search bar to find definitions for dense terminology like "UTXO", "CoinJoin", or "Address Reuse".
  3. Click 'X' to close the glossary when you're done studying.

2. Analysis Heuristics

Sherlock applies the following 9 heuristics to every transaction parsed. These behaviors help group addresses and track fund flows across the pseudonymous blockchain.

Common Input Ownership (CIOH)

  • What it is: Multiple inputs consumed in a single transaction.
  • Why it matters: It implies all inputs are controlled by the same wallet, since signing them requires all corresponding private keys.

Change Detection

  • What it is: Identifying which output is the internal "change" returned to the sender.
  • How it works: Sherlock looks for outputs that match the input script type, or eliminates outputs that look like human-entered round numbers (e.g. 0.05 BTC).

Consolidation

  • What it is: Combining many UTXOs into one or two outputs.
  • Why it matters: Wallets do this automatically for housekeeping. It strongly signals that all inputs belong to the same entity.

Address Reuse

  • What it is: The exact same address appears in both inputs and outputs.
  • Why it matters: It deterministically links multiple UTXOs to the same public key, destroying privacy.

Round Number Payments

  • What it is: Outputs with exact, round values (e.g. 0.01 BTC).
  • Why it matters: Humans type round numbers to pay people. Non-round numbers are usually auto-generated change.

CoinJoin

  • What it is: Users combine inputs to create many identical equal-value outputs.
  • Why it matters: It breaks the CIOH assumption deliberately to mix funds and preserve privacy.

OP_RETURN Data

  • What it is: Provably unspendable outputs containing arbitrary hex data.
  • Why it matters: Indicates a timestamp, notarization, or secondary layer protocol rather than a financial transfer.

Self-Transfer

  • What it is: All inputs and outputs share the exact same script type with no obvious payment.
  • Why it matters: Usually a user just moving their own funds around for security or privacy rotation.

Peeling Chain

  • What it is: A large balance is spent into a small payment and a large change output, repeatedly over time.
  • Why it matters: Often reveals an exchange withdrawal system or an entity slowly distributing funds.
